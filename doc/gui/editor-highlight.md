# Editor styling: find/replace and runtime syntax highlighting

## Scope

Two related features sit on the same foundation in `bd_widget_editor`:

1. Find/replace highlighting (built).
2. Runtime-configurable syntax highlighting (built; `bd_syntax.{c,h}`).

Both need the editor to override colour, weight, and background over arbitrary
byte ranges, and both must coexist. That shared substrate is a layered style-run
list, described first, then the find/replace feature, then the syntax design.

## The shared substrate: layered style runs

The editor already carried styling as a flat list of runs, each a byte range
plus a `bd_rich_style` (flags, fg, bg). The renderer segments each line by run
and draws per-run colour, background, bold, italic, underline, and strike. The
drawing half of both features already existed. The gap was that every run lived
in one undifferentiated list, so a syntax pass, a find highlight, and a
transient mark would clobber each other, and `bd_editor_clear_styles` erased all
of them at once.

Each run now carries a layer:

```c
enum {
    E_LAYER_SYNTAX = 0,  /* regenerated by a syntax pass */
    E_LAYER_USER,        /* app style_span / highlight_* */
    E_LAYER_FIND,        /* every find match */
    E_LAYER_MARK,        /* the current find match / a transient mark */
    E_N_LAYERS,
};
```

Composition (`e_style_at`) resolves overlaps attribute by attribute rather than
"last run wins": flags OR together, while `fg` and `bg` each take the value from
the highest-layer run that sets them (zero means inherit). So a find highlight,
which sets only `bg` on a high layer, washes the background without erasing the
syntax foreground colour underneath. A layer can be cleared on its own
(`e_clear_layer`), which is what lets a syntax pass regenerate on every keystroke
without disturbing find highlights, and vice versa.

This is the whole reason to do find/replace first: it forces the layering that
syntax highlighting then reuses for free.

## Find/replace (implemented)

Public API (`bd_widget_editor.h`):

```c
enum { BD_FIND_ICASE = 1 << 0, BD_FIND_WORD = 1 << 1 };
int  bd_editor_find(bd_id, const char *needle, unsigned flags);  /* -> count */
int  bd_editor_find_next(bd_id);   /* -> current index, wraps */
int  bd_editor_find_prev(bd_id);
int  bd_editor_find_count(bd_id);
int  bd_editor_find_current(bd_id);
void bd_editor_find_clear(bd_id);
int  bd_editor_replace(bd_id, const char *repl);      /* current match, re-finds */
int  bd_editor_replace_all(bd_id, const char *repl);  /* -> number replaced */
```

`bd_editor_find` scans the buffer, records each match offset, adds a FIND-layer
background wash for every match, marks the first match at or after the caret as
current (a brighter MARK-layer wash), and points the caret at it so the existing
"keep the caret in view" logic scrolls it on-screen. `find_next`/`find_prev`
cycle the current match with wraparound. Matching is byte-literal with optional
ASCII case-insensitivity and whole-word bounds (`[A-Za-z0-9_]`-delimited).
`replace` splices the current match and rescans (offsets shift); `replace_all`
splices from last match to first so earlier offsets stay valid, then clears the
find. Matches are non-overlapping.

The highlight is static between calls: it survives edits because run offsets
shift with the text, but new or removed matches are not reflected until the next
`bd_editor_find`. A find UI re-calls `bd_editor_find` on each keystroke in its
field, which is cheap at this editor's scale.

## Runtime syntax highlighting (implemented)

### The reference (compact-pascal playground)

The compact-pascal web playground highlights with a **JSON-configured state
machine tokenizer**, not regexes or match rules. A language is a set of named
states; each state has ordered rules that match a character class and transition
to another state:

```json
"idle": {
  "style": "idle",
  "rules": [
    { "match": "a-zA-Z_", "next": "ident", "buffer": true },
    { "match": "{",       "next": "comment", "recolor": 1 },
    { "match": "'",       "next": "string",  "recolor": 1 }
  ]
},
"ident": { "defaultNoeat": true,
  "keywords": { "program": "keyword", "begin": "keyword", "end": "keyword" } }
```

The tokenizer walks the buffer one character at a time, carrying the current
state. For each character it picks the first rule whose class matches, assigns
the state's style to the character, optionally recolours a few preceding
characters (so an opening `{` gets the comment colour), and moves to the next
state, optionally without consuming the character (`noeat`) so the same
character is re-processed in the new state. Identifiers buffer until they end,
then a keyword map recolours the whole word. Because the state persists across
newlines, multi-line comments and strings fall out naturally.

Notable properties, and their fit here:

- **State machine, not regex.** Portable and small. Good fit for C.
- **State carried across lines.** The correct way to handle block comments and
  strings. Keep it.
- **`recolor` / `noeat` / keyword-on-exit.** Compact ways to colour delimiters
  and keywords in a single pass. Worth adapting.
- **Per-character style array, rendered to HTML `<span>`s.** A browser idiom. In
  C we emit `(start, end, style)` spans into the SYNTAX layer instead.
- **Full re-tokenize from offset 0 on every keystroke.** Fine for
  playground-sized text. For a large buffer, cache the parse state at line
  starts and resume from the edited line. Not needed at birdie's scale on day
  one; note it as the known optimization.
- **JSON config, fetched at load, keyed by file extension.** The config *shape*
  (state machine) is the valuable part; JSON specifically is not required.

### C design (as built; see `bd_syntax.h` for the exact API)

Split the work in two, mirroring what find/replace already established:

1. A standalone engine, `bd_syntax`, that turns text + a language definition
   into a list of `(start, end, style)` spans. It owns no widget state. This is
   a direct C translation of the state machine above:

   ```c
   typedef struct bd_syntax_style { unsigned flags; uint32_t fg, bg; } bd_syntax_style;

   typedef struct bd_syntax_rule {
       const char *match;   /* char class: "a-zA-Z0-9_", "{", "\n" */
       int   next;          /* target state index */
       int   recolor;       /* restyle this many preceding chars */
       int   noeat;         /* re-process the char in `next` */
       int   buffer;        /* begin accumulating for a keyword lookup */
   } bd_syntax_rule;

   typedef struct bd_syntax_state {
       int style;                 /* style index for chars in this state */
       int deflt;                 /* fallback transition */
       int default_noeat;         /* re-process the terminator in `deflt` */
       const bd_syntax_rule *rules; int nrule;
       const struct { const char *word; int style; } *keywords; int nkw;
   } bd_syntax_state;

   typedef struct bd_syntax_lang {
       const char *name;
       int start_state;
       const bd_syntax_style *styles; int nstyle;   /* style index -> appearance */
       const bd_syntax_state *states; int nstate;
   } bd_syntax_lang;

   int bd_syntax_run(const bd_syntax_lang *, const char *text, int len,
                     bd_syntax_span *out, int max);   /* -> span count */
   ```

2. An editor hook that owns re-highlighting into the SYNTAX layer:

   ```c
   void bd_editor_set_syntax(bd_id, const bd_syntax_lang *);   /* NULL = off */
   ```

   On each edit the editor clears its SYNTAX layer, runs the engine over the
   buffer, and adds one SYNTAX-layer run per span. Find highlights, app styling,
   and transient marks are untouched because they live on other layers.

### The config source (decided)

The engine is data-driven; the decision was where a `bd_syntax_lang` comes from.
Rather than a single source, the shipped design does three things: it **parses a
definition from an in-memory text buffer** (`bd_syntax_parse`, so a language can
come from a compiled-in string, a file the app reads, or anywhere), it **compiles
in useful defaults** (`bd_syntax_builtin("lua")` / `("abc")`, lazily parsed and
cached), and it offers an **override + autodetect** API so an app can force a
language or let the extension pick one:

```c
bd_syntax_lang *bd_syntax_parse(const char *text, int len);   /* from memory */
const bd_syntax_lang *bd_syntax_builtin(const char *name);    /* "lua","abc" */
const bd_syntax_lang *bd_syntax_for_name(const char *file);   /* by extension */
void bd_syntax_register(const char *name, const char *const *exts,
                        const bd_syntax_lang *);              /* extend detect */
void bd_editor_set_syntax(bd_id, const bd_syntax_lang *);     /* override/off */
```

JSON was deliberately not adopted: it would add a third-party parser to a toolkit
that vendors only stb. A Lua-defined language was rejected for the shippable
toolkit for the same coupling reason (the app could still register one). The
text format is flat and line-oriented, parsed from memory with no file I/O in
the toolkit:

```
name lua
start idle
# style <name> <#RRGGBB|#RRGGBBAA|-> [bold] [italic] [underline] [bg:#RRGGBB]
style default -
style keyword #E0A030 bold
style comment #6A9955 italic
# state <name> [style <s>] [default <state>] [noeat]
#   rule <charclass> <next> [recolor[N]] [noeat] [buffer]
#   str  <literal>   <next> [recolor] [noeat]
#   keyword <style> <word>...
state idle style default default idle
  rule a-zA-Z_ ident buffer
  rule 0-9 number recolor
  str -- comment recolor
state ident style default default idle noeat
  rule a-zA-Z0-9_ ident
  keyword keyword function local end if then else return
```

`rule` matches one byte against a character class (`a-zA-Z0-9_`, ranges and
literals, `\n \t` escapes); `str` matches a fixed multi-byte literal (so `--`,
`[[`, `<!--` are one transition). `recolor[N]` restyles the last N bytes (a
`str` recolours its whole literal) with the target state's style, colouring the
opening delimiter as comment/string. `noeat` re-processes the byte in the next
state; a state marked `noeat` re-processes the terminator through its `default`
and, if it has a `keyword` map, looks the buffered word up on the way out.
`buffer` marks where a keyword word starts. State persists across newlines, so an
unterminated comment or string simply continues.

### Editor integration

The editor holds a `const bd_syntax_lang *` and a dirty flag set by every buffer
edit. At the top of render, if dirty, it clears its SYNTAX layer, runs the engine
over the whole buffer once, and adds one SYNTAX-layer run per span, then clears
the flag. So highlighting is at most one re-tokenize per frame regardless of how
many keystrokes landed, and find highlights, app styling, and marks are never
touched. `bd_editor_set_syntax(id, lang)` installs or clears (NULL) the language;
an app typically calls `bd_syntax_for_name(filename)` and passes the result.


### Performance

Full re-tokenize on every edit is O(n) and fine for the small script, trigger,
and notation buffers birdie edits. If a large-buffer use appears, cache the
engine's state at each line start and re-run only from the edited line forward.
The span output already lands in the SYNTAX layer, so incremental updates change
only the engine, not the editor or the renderer.
